# Deliverance 0.0.10 Release Notes

Deliverance 0.0.10 is a large release. The short version: Qwen3 landed, JQ4/quantize-on-demand became a first-class workflow, Grace became the tokenizer path to care about, tensor-parallel moved from sketches into real infrastructure, and the project gained serious benchmarking, streaming, tool-calling, and local-agent support.

This release also includes substantial internal refactoring of generation, model inputs, KV cache behavior, native kernels, and web/client APIs. Some commit messages in this range were intentionally terse, so these notes summarize the actual user-facing and developer-facing changes from `v0.0.9` to `v0.0.10`. Post-tag fixes are intentionally excluded.

## Highlights

- Added Qwen3 model support, including thinking/non-thinking prompt behavior.
- Added quantize-on-demand for local JQ4 model creation and reuse.
- Published and switched benchmark defaults to uploaded Qwen3 JQ4 Hugging Face repos.
- Added Gemma 4 experimental model support and prompt/tool parsing work.
- Added tensor-parallel generation infrastructure with HTTP and Netty transports.
- Added Grace tokenizer improvements and a tokenizer benchmark against Hugging Face tokenizers.
- Added native SIMD SAXPY and broadened native tensor parity testing.
- Added WebGPU/Dawn GPU output-projection support for targeted Q4 acceleration.
- Added nanocode-deliverance, a small local terminal coding agent backed by Deliverance HTTP.
- Added vibrant-maven-plugin back into the build with chat/shell workflows.
- Added a generated OpenAPI client module and broader web streaming/tool-call support.
- Added benchmark scripts, JSONL transcripts, profile-stage output, model-config JSON, and a thinking smoke benchmark.

## Qwen3 And Thinking Mode

0.0.10 adds a Qwen3 model family implementation:

- `Qwen3Config`
- `Qwen3Model`
- `Qwen3ModelType`
- `Qwen3CausalSelfAttention`
- Qwen3 prompt/template support
- Qwen3 tool-call parser support
- Qwen3 HF-shape/config tests
- Qwen3 small-model integration tests

Qwen3 can now run in both thinking and non-thinking modes through prompt-template arguments such as `enable_thinking`. This is used by tests, benchmarks, nanocode, and the web/chat stack.

The release also adds a qualitative thinking smoke benchmark, `ThinkingSmokeBenchmark`, which runs bounded prompts and records:

- pass/fail
- finish reason
- generated tokens
- time
- time-to-first-token
- full response text

This benchmark is meant to catch obvious thinking/tool regressions after model/runtime changes. It is not a throughput benchmark.

The release now assumes Qwen3 JQ4 models can come from Hugging Face instead of only local quantize-on-demand output:

## Quantize On Demand And JQ4

Quantize-on-demand became a real workflow in 0.0.10. The builder can create a quantized target model if it is missing, then reuse it on later loads:

```java
AutoModelForCausaLm.newBuilder(fetcher)
    .withQuantizeOnDemand(DType.Q4, "owner", "model-JQ4")
    .buildLocalTransformerModel();
```

The quantizer now writes useful provenance alongside generated model artifacts:

- `deliverance-quantization.json`
- updated generated `README.md`
- `.finished` completion marker

The release includes docs for:

- quantize-on-demand
- JQ4 tensor format
- tensor engines and JQ4
- native SIMD kernels
- GPU output projection

## Grace Tokenizer Work

0.0.10 moves the project further toward `grace.PreTrainedTokenizer` as the runtime tokenizer path. The older bespoke tokenizer classes were removed from core model packages, and Grace gained more complete tokenizer behavior for modern model families.

Notable tokenizer work:

- Llama tokenizer support in Grace
- BERT tokenizer support in Grace
- byte-level BPE improvements
- Qwen/Gemma tokenizer fixture and behavior tests
- prompt-template contract tests modeled after Hugging Face behavior
- tokenizer local fixture tests

The release also adds a tokenizer showdown benchmark:

- Java Grace tokenizer benchmark
- Python Hugging Face fast-tokenizer benchmark
- shared tokenizer-showdown corpus
- chart and writeup

The measured local result was roughly a `1.66x` Grace throughput win on the benchmark corpus while preserving token-id parity for the tested input.

## Gemma 4 Experimental Support

Gemma 4 support is present but still marked as experimental. This release adds:

- Gemma4 config parsing
- model class and model type
- Gemma4 attention and transformer block variants
- Gemma4 PLE support
- response parser
- tool-call parser
- prompt integration tests
- exploratory reasoning/prompt tests
- HF-style ported tests for text-model behavior

Gemma4 is useful for development, but not all reasoning behavior is considered solved. Tests that require locally cached Gemma4 quantized weights now use assumptions so the suite skips cleanly if the model directory is absent.

New pieces include:

## Prefix Cache And KV Cache Work

0.0.10 includes major KV-cache cleanup and prefix-cache testing.

The prefix cache now has stronger mechanical tests around:

- block-aligned lookup
- cache salt isolation
- KV row copy round trips
- key/value preservation after cache hit

The disk KV backend was documented and received sweeper support for stale page cleanup. Disk-backed pages are active KV pages, not a durable content-addressed prefix-cache store. The release notes and docs call out this distinction because naive persistent prefix snapshots can grow very quickly.

## Native SIMD And Tensor Operations

Native tensor support expanded significantly.

New native SIMD paths include:

- F32 SAXPY
- batched F32 SAXPY
- BF16/Q4 native parity tests
- Gemma4-shaped Q4 projection parity tests
- broader Q4 row/offset handling

Panama tensor operations also received fixes and broader coverage, including GEMM tail handling and Q4/BF16 parity cases.

Tensor shape and sparse offset handling were improved to support row-sharded/tensor-parallel weights and packed Q4 layouts more reliably.

## WebGPU / Dawn GPU Work

Deliverance gained an experimental WebGPU/Dawn tensor provider and targeted GPU acceleration for Q4 output projection.

This is not full GPU inference yet. The useful path in this release is targeted GPU output projection, especially for large Q4 output heads. The docs include a measured example where sampler output projection time dropped materially for Qwen3-4B-JQ4.

The GPU path includes:

- Dawn/WebGPU native library download support
- GPU JNI bridge
- WGSL GEMM shaders
- GPU tensor registration
- runtime singleton initialization
- fallback behavior for unsupported shapes/offsets
- GPU output projection docs and chart

## Nanocode Deliverance

0.0.10 adds `nanocode-deliverance`, a tiny terminal coding agent backed by a local Deliverance HTTP server.

It includes local tools:

- `read`
- `write`
- `edit`
- `glob`
- `grep`
- `java_sandbox`
- optional `bash` when explicitly enabled

Nanocode supports streaming chat completions and accumulates streamed tool-call deltas. It also has a single JSON config file instead of a pile of CLI switches or environment variables.

Session commands include:

- `/config get rounds`
- `/config set rounds 3`
- `/config get thinking`
- `/config set thinking off`
- `/config help`
- `/c` to clear context
- `/q` or `exit` to quit

The module also ships Qwen3/Llama/TinyLlama client configs and server launch scripts.

## Breaking Or Notable Behavioral Changes

- Core model tokenizers were removed in favor of the Grace tokenizer path.
- The generation stack was refactored; downstream code using internals may need updates.
- The web client is now generated from the OpenAPI spec in a dedicated `client` module.
- Some large-model tests are guarded by assumptions and require local cached model directories.
- Gemma4 is present but experimental.
- GPU acceleration is experimental and targeted; full GPU inference is not the recommended default path.
- The prefix cache remains in-memory for prompt-prefix reuse; disk-backed active KV pages are not a durable shared-prefix store.

## Known Limits

- Gemma4 thinking behavior is not considered fully solved.
- WebGPU/Dawn support depends on platform native libraries and still falls back for unsupported shapes.
- Native classifier artifacts are platform-specific and can conflict if copied between platforms.
- Long-context prefix caching needs further work before disk persistence is safe for production-scale sessions.
- The thinking smoke benchmark is a smoke test, not a rigorous evaluator.

## Upgrade Notes

- Use JDK 25 for most modules.
- Use `-am` when building modules that depend on local snapshots.
- Rebuild the `web` package after core changes before running server scripts; Spring Boot repackaging happens at `package` time.
- For Qwen JQ4 benchmarks and web configs, prefer the uploaded Hugging Face repos:
  - `edwardcapriolo/Qwen3-0.6B-JQ4`
  - `edwardcapriolo/Qwen3-4B-JQ4`
